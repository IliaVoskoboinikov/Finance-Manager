# Модуль Auth и Сетевая Архитектура

Модуль `auth` отвечает за аутентификацию пользователей, управление сессиями (JWT) и координацию доступа к защищенным ресурсам приложения.

---

## 1. Архитектура Сессий

Приложение использует три состояния авторизации, определенных в `AuthStatus`:

*   **AUTHORIZED**: Пользователь вошел в систему. Доступен полный функционал синхронизации.
*   **GUEST**: Полноценный офлайн-режим. Данные сохраняются только локально. Сетевые запросы к бизнес-логике блокируются.
*   **UNAUTHORIZED**: Состояние до входа или после инвалидации сессии. Доступны только экраны логина/регистрации.

### Источник Истины (Single Source of Truth)
`AuthManager` является центральным компонентом, который:
1. Кэширует `AccessToken` в памяти (`@Volatile`) для мгновенного доступа.
2. Синхронизирует параллельные запросы на обновление токенов через `Mutex`.
3. Координирует очистку данных при инвалидации сессии.

---

## 2. Работа с Сетью и JWT

### Система "Двойного Retrofit"
Для предотвращения взаимных блокировок (Deadlocks) при обновлении токенов, сетевой слой разделен на два клиента:

1.  **Main Retrofit**: Используется для всех бизнес-запросов.
    *   Имеет `AuthInterceptor` (добавляет Bearer токен).
    *   Имеет `TokenAuthenticator` (автоматически вызывает `/refresh` при 401 ошибке).
    *   Имеет `GuestAccessInterceptor` (блокирует выход в сеть для гостей).
2.  **Base (Auth) Retrofit**: Используется исключительно для запросов `/login`, `/register`, `/refresh`.
    *   **Не имеет** аутентификатора и интерцепторов авторизации. Это гарантирует, что если сам запрос на обновление токена получит 401, система не уйдет в бесконечную рекурсию.

### Алгоритм Обновления Токенов
1. Обычный запрос получает `401 Unauthorized`.
2. `TokenAuthenticator` перехватывает ответ и вызывает `AuthManager.refreshTokenIfNeeded(oldToken)`.
3. `AuthManager` захватывает `Mutex`. Если другой поток уже обновил токен, возвращается актуальный из кэша.
4. Выполняется запрос `/auth/refresh` через "чистый" `Base Retrofit`.
5. При успехе: новые токены сохраняются в зашифрованный `DataStore` и кэш.
6. При ошибке 401/403: инициируется полная очистка данных (защита приватности).
7. При ошибке сети: пробрасывается `IOException`, сессия сохраняется.

---

## 3. Безопасность и "Сетевой Предохранитель"

Для защиты данных и предотвращения лишних ошибок используется `GuestAccessInterceptor`.
*   **Логика**: Если пользователь находится в статусе `GUEST` или `UNAUTHORIZED`, любые попытки сетевого взаимодействия (кроме путей `/auth/*`) прерываются исключением `GuestModeException`.
*   **Обработка**: `safeApiCall` распознает это исключение и не логирует его как системную ошибку, возвращая нейтральный результат "Network Error".

---

## 4. Управление Данными и Очистка

Приложение реализует **селективную очистку** базы данных через `DatabaseCleanupManager`.

### Правила очистки:
*   **Удаляются**: Транзакции, Счета, Лимиты.
*   **Сохраняются**: Категории (системные предзаполненные данные).

### Сценарии удаления:
1.  **Автоматически (401)**: Если refresh-токен протух, данные удаляются мгновенно для защиты приватности.
2.  **Ручной Logout**: Пользователь выбирает в диалоге: "Очистить всё" или "Оставить локально (перейти в режим гостя)".
3.  **Login/Register**: Если в базе есть "гостевые" данные, пользователь выбирает: "Синхронизировать с аккаунтом" или "Начать с чистого листа".

## 5. UI и Синхронизация

### Реактивная Навигация
`RootNavGraph` подписан на `AuthStatus`. Как только статус меняется на `UNAUTHORIZED` (например, из-за 401 ошибки глубоко в сетевом слое), навигация мгновенно сбрасывает стек и переводит пользователя на экран логина.

### Принудительная Синхронизация
При входе в аккаунт (`AUTHORIZED`) приложение инициирует блокирующую синхронизацию через `SyncCoordinator`:
1. Показывается полноэкранный оверлей загрузки.
2. Приложение дожидается завершения `syncAll()`.
3. В случае успеха пользователь видит актуальные данные.
4. В случае сбоя сети показывается уведомление, но пользователь допускается к локальным данным.
